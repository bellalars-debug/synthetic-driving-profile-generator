# Clustering Plan

Status: design only — no code written yet. This document specifies how
`data/processed/employee_features.parquet` (the `build_features.py` output:
one row per `HOUSEID`+`PERSONID`) should be turned into a small set of
representative employee travel-behavior clusters. It does not implement
anything; see `docs/feature_engineering_plan.md` for how the input columns
referenced below were derived.

---

## 1. Clustering objective

The goal is to group individual employee driving-behavior profiles into a
small number of **representative travel behavior archetypes** — e.g. "short
commute, onsite, dedicated vehicle" vs. "long commute, hybrid schedule,
shared household vehicle" — that can stand in for the full NHTS sample when
generating a synthetic employee population.

This is explicitly a means to an end, not the end itself. NHTS gives us
~1 surveyed day for a fixed, non-representative set of real people; no
individual real employee's exact record should appear in a synthetic
workforce (privacy and generalizability), and a downstream generator needs
*distributions to sample from*, not a fixed list of rows. Clusters solve
both problems: each cluster defines a coherent sub-population with its own
joint feature distribution, and synthetic employees are generated by
sampling from a cluster's distribution rather than resampling real rows
directly (see §7).

Consequently, the clusters should be chosen to separate people primarily
along dimensions that drive **workplace parking/charging opportunity** —
whether and how far someone commutes, when they arrive and leave, how
"available" their vehicle is — not along dimensions that just happen to
have high variance in NHTS (e.g. age in years). This objective is what
drives feature selection in §3.

---

## 2. Input dataset

- **Source:** `data/processed/employee_features.parquet`, produced by
  `build_features.py`.
- **Grain:** one row per `HOUSEID`+`PERSONID` — a single person's single
  surveyed weekday, already aggregated up from trip-level records (plan:
  `docs/feature_engineering_plan.md` §1). No further aggregation is needed
  before clustering; each row is already the unit clustering should operate
  on.
- **Population filter:** the feature table as built includes every person
  in the cleaned weekday trip data, workers and non-workers alike
  (`is_worker` distinguishes them). Clustering should run on **workers with
  an observed commute that day** (`is_worker == True` and
  `work_trip_count > 0`), since the objective (§1) is specifically about
  workplace charging opportunity — a non-worker or a worker who didn't
  travel to work that day has no workplace dwell window to characterize,
  and their commute-derived columns (`commute_distance_survey_miles`,
  `work_arrival_time`, `work_departure_time`, …) are structurally NaN (not
  applicable), not a value clustering can meaningfully group on. Non-worker
  rows should still be kept in the parquet file for other project uses
  (e.g. household-level context), just excluded from the clustering input
  matrix itself.
- **Key columns available on that filtered population** (from
  `create_employee_feature_table`): `age`, `age_band`, `worker_status`,
  `household_income_bracket`, `household_size`, `household_vehicle_count`,
  `work_trip_count`, `commute_distance_survey_miles`,
  `commute_distance_trip_miles`, `commute_duration_minutes`,
  `work_arrival_time`, `work_departure_time`, `trips_per_day`,
  `total_daily_miles`, `total_driving_minutes`,
  `average_trip_distance_miles`, `number_of_stops`, `vehicles_per_driver`,
  `vehicle_per_driver_adequate`, `used_household_vehicle`,
  `household_vehicle_trip_count`. `HOUSEID`/`PERSONID` are identifiers, not
  features, and must be excluded from the clustering matrix but retained
  as an index for joining cluster labels back onto the source table.

## 3. Feature selection for clustering

### Recommended feature groups

| Group | Columns | Role in clustering |
|---|---|---|
| **Commute behavior** | `commute_distance_survey_miles` (preferred over `commute_distance_trip_miles` — see note below), `commute_duration_minutes`, `work_arrival_time`, `work_departure_time` | **Primary.** These most directly determine the workplace charging window: when the vehicle arrives, how long it stays, how much range the commute itself consumed. This is the group that should do most of the work separating clusters. |
| **Daily mobility** | `trips_per_day`, `total_daily_miles`, `total_driving_minutes`, `number_of_stops` | **Primary.** Captures whether the day is a direct commute or a fragmented, multi-stop day — directly affects whether the workplace dwell window is one continuous session or interrupted. |
| **Vehicle availability** | `vehicles_per_driver`, `vehicle_per_driver_adequate`, `used_household_vehicle` | **Primary.** Determines whether "the commute vehicle" is a stable, dedicated concept for this person or a contended, shared resource — directly relevant to whether a charging session can be reliably planned around this person. |
| **Demographics** | `age_band`, `worker_status` (constant `True` after the §2 filter) | **Secondary / contextual.** Useful for describing *who* is in a cluster and for calibrating a realistic synthetic population, but age and worker status have no direct mechanism that changes workplace charging opportunity — a 30-year-old and a 50-year-old with the same commute distance and arrival time pose the same charging problem. |
| **Household variables** | `household_income_bracket`, `household_size`, `household_vehicle_count` | **Secondary / contextual.** Same reasoning as demographics: these shape *which kinds of people* end up in a cluster (useful for the synthetic-population realism check in §7) but don't themselves determine charging-relevant timing. `household_vehicle_count` is largely redundant with `vehicles_per_driver`, which already normalizes it by driver count — prefer the ratio. |

### Features that should NOT dominate clustering

- **`age`** (continuous, not `age_band`) should not be used directly.
  Distance-based clustering methods (KMeans, hierarchical with Euclidean
  linkage) are sensitive to a feature's raw numeric range and variance;
  age spans a genuinely wide range (roughly 16–90) with no strong
  behavioral gradient over most of that range, so an unscaled `age` column
  can dominate the distance metric purely because of its spread, not
  because it's more informative than commute distance or arrival time. If
  age is included at all, use the coarser `age_band` as a categorical
  feature (see §4) rather than a raw scaled numeric.
- **`household_income_bracket`** is an ordinal bracket code, not a
  continuous dollar amount — it must be treated as ordinal/categorical
  (§4), not scaled as if it were a smooth numeric variable, or its
  arbitrary bracket-code spacing will distort distances.
- **Identifiers and near-identifiers** — `HOUSEID`, `PERSONID`,
  `work_trip_count` (nearly constant at 1 within the filtered population,
  carries almost no separating signal) should be excluded entirely rather
  than merely down-weighted.
- **Redundant/derived pairs** — including both a raw count and its
  normalized ratio (e.g. `household_vehicle_count` alongside
  `vehicles_per_driver`, or both `commute_distance_survey_miles` and
  `commute_distance_trip_miles`) lets one underlying concept count twice
  in a distance metric, implicitly over-weighting it relative to
  single-column concepts like `number_of_stops`. Pick one representative
  per concept for the clustering matrix; keep the other in the source
  table for validation/cross-checking only.  `commute_distance_survey_miles`
  (`GCDWORK`) is preferred over the trip-based sum because it is complete
  for every worker with a `GCDWORK` response, while the trip-based sum is
  NaN whenever `WHYTRP1S` didn't tag a leg as work-purpose that day even if
  the person worked.

## 4. Preprocessing

### Missing values

Missingness in this table is not uniform in meaning, and should be handled
differently depending on why a value is missing:

- **Structural "not applicable" NaNs**, e.g. commute columns for a
  non-worker or a worker with `work_trip_count == 0` — resolved by the §2
  population filter, not imputation. These rows shouldn't reach the
  clustering step with a value to impute in the first place.
- **"No driving trips that day" NaNs**, e.g. `total_daily_miles`,
  `total_driving_minutes`, `average_trip_distance_miles` when a worker
  commuted by a non-driving mode (transit, walk, bike). This is a real,
  meaningful zero-driving-behavior case, not missing data — impute as `0`
  (or carry an explicit `commuted_by_driving_mode` flag derived from
  whether these are NaN, if that distinction itself is useful context for
  a cluster) rather than dropping the row or median-imputing, which would
  fabricate driving behavior that didn't happen.
- **True missing/unknown survey responses**, e.g. `household_income_bracket`
  or `age`/`age_band` when NHTS recorded a refusal or "not ascertained"
  code (already converted to NaN by `clean.py`). For a first pass, impute
  numeric fields with the median and categorical fields with an explicit
  `"unknown"` category (rather than the mode) so an unknown response isn't
  silently folded into whatever category happens to be most common —
  that would bias cluster composition toward the majority category.
- **`vehicles_per_driver` NaN** (household has 0 drivers, an edge case)
  should be dropped from the clustering population rather than imputed —
  it indicates a data anomaly for a person who is coded as a worker with a
  commute, not a typical case to represent with a filled-in value.

### Categorical encoding

- **Nominal categoricals** (`worker_status` — though this is constant
  post-filter and can simply be dropped; any future mode/location
  categoricals reintroduced from `docs/feature_engineering_plan.md`, e.g.
  `WRKLOC`, `WRKTRANS`) should use one-hot encoding for KMeans/hierarchical
  clustering, since these methods interpret numeric closeness as
  similarity and an arbitrary integer code (e.g. mode 1 vs. mode 6) has no
  meaningful distance relationship.
- **Ordinal categoricals** (`age_band`, `household_income_bracket`) should
  be encoded as integers reflecting their natural order (e.g. `<25`→0,
  `25-34`→1, …) rather than one-hot, so that clustering can use the
  meaningful "closer in rank" relationship between adjacent bands instead
  of treating every band as equally dissimilar from every other.
- **Booleans** (`vehicle_per_driver_adequate`, `used_household_vehicle`)
  pass through as 0/1 with no encoding step needed.

### Numerical scaling

All continuous features (`commute_distance_survey_miles`,
`commute_duration_minutes`, `work_arrival_time`, `work_departure_time`,
`trips_per_day`, `total_daily_miles`, `total_driving_minutes`,
`number_of_stops`, `vehicles_per_driver`) must be standardized (zero mean,
unit variance) before any distance-based clustering (KMeans, hierarchical,
and GMM with a diagonal/spherical covariance assumption all effectively
compare feature scales). Two scale mismatches in this data make this a
hard requirement, not a nice-to-have:

- `work_arrival_time`/`work_departure_time` are encoded HHMM-style (e.g.
  `830.0` for 8:30am, per `build_features.py`), so their raw numeric range
  (roughly 0–2400) dwarfs a feature like `vehicles_per_driver` (roughly
  0–3) or `number_of_stops` (typically 0–10) by orders of magnitude.
  Left unscaled, arrival/departure time alone would decide almost every
  cluster boundary.
- `total_daily_miles` and `commute_distance_survey_miles` have long right
  tails (a small number of very long commutes/driving days). Standard
  z-score scaling is a reasonable default; if exploratory analysis shows
  those tails are heavily skewed enough to distort even scaled distances,
  a log1p transform before scaling is worth evaluating (defer this
  decision to implementation, based on the actual distribution).

## 5. Candidate clustering methods

| Method | How it fits this problem | Strengths here | Weaknesses here |
|---|---|---|---|
| **KMeans** | Partition employees into *k* groups by minimizing within-cluster distance to a centroid. | Fast, scales well to the full NHTS sample size, simple to explain to non-technical stakeholders ("k typical employee types"), centroids are directly interpretable as an "average archetype" — convenient for the sampling step in §7. Good default starting point. | Assumes roughly spherical, similarly-sized clusters and is sensitive to outliers (e.g. very long commutes) and to the scaling choices in §4; requires *k* to be chosen upfront (§6); only handles continuous/encoded-numeric input, so ordinal/one-hot encoding (§4) is mandatory first. |
| **Hierarchical clustering** (agglomerative, e.g. Ward linkage) | Builds a dendrogram of nested groupings rather than a single flat partition. | Doesn't require committing to *k* upfront — the dendrogram can be cut at different heights to compare granularities (e.g. "3 broad archetypes" vs. "8 fine-grained ones"), which is useful for exploring this dataset before committing to a production *k*. Also useful diagnostically even if KMeans is used for the final model, since the dendrogram can validate that KMeans's chosen *k* is a sensible cut point. | Doesn't scale as well computationally (naive implementations are roughly O(n²)–O(n³)) — likely fine for the NHTS employee subsample size but worth checking; like KMeans, sensitive to the same scaling/outlier issues, and Ward linkage carries the same "roughly spherical cluster" assumption. |
| **Gaussian Mixture Models (GMM)** | Probabilistic clustering: models the population as a mixture of Gaussians and assigns each employee a soft (probabilistic) cluster membership. | Soft assignment is a genuine advantage for §7's downstream use — sampling synthetic employees from "cluster 2's distribution" is a more natural fit with a GMM's fitted per-cluster Gaussian than with a hard KMeans partition, since the GMM already *is* a generative distribution per cluster. Can also model elliptical (not just spherical) cluster shapes depending on the covariance type chosen. | More parameters to fit (covariance structure per cluster) means it needs more data per cluster to fit stably and is more prone to overfitting with too many clusters; less immediately interpretable than KMeans centroids; still needs a chosen number of components and, like the others, needs the scaling/encoding from §4 done first. |

**Recommendation for a first pass:** start with KMeans for its simplicity
and interpretability, use hierarchical clustering (dendrogram) as a
diagnostic to sanity-check the chosen *k* and confirm clusters are not an
artifact of KMeans's spherical assumption, and evaluate GMM as a follow-up
specifically because its native generative/probabilistic structure maps
cleanly onto the sampling step described in §7 — if KMeans's hard
archetypes prove too rigid for realistic synthetic sampling, GMM is the
natural upgrade path rather than a from-scratch redesign.

## 6. Choosing the number of clusters

No single metric should be trusted alone; use a combination and reconcile
them against domain judgment:

- **Elbow method** — plot within-cluster sum of squared distances
  (inertia) against *k* and look for the point of diminishing returns.
  Simple to compute for KMeans but the "elbow" is often ambiguous in real
  data, so it should be a starting hypothesis, not the deciding vote.
- **Silhouette score** — measures how well-separated and internally
  cohesive clusters are (ranges -1 to 1, higher is better); more
  discriminating than the elbow method when the elbow plot is ambiguous,
  and directly comparable across candidate *k* values.
- **Gap statistic** — compares within-cluster dispersion against a null
  reference distribution (data with no real cluster structure); a
  principled way to check whether adding another cluster is capturing real
  structure or just fitting noise, more rigorous than eyeballing an elbow
  plot.
- **BIC/AIC** (for GMM specifically) — penalizes model complexity, giving
  a principled way to compare different numbers of Gaussian components
  without simply favoring more components every time.
- **Domain interpretability** — after the quantitative methods above
  suggest a reasonable range (e.g. "somewhere between 4 and 8"), the
  deciding factor should be whether each resulting cluster corresponds to
  a *distinct, describable employee travel pattern* that a non-technical
  stakeholder could recognize (e.g. "short urban commuter," "long
  suburban commuter with a shared vehicle," "hybrid/part-time worker").
  A statistically optimal *k* that produces two clusters differing only in
  a 5-minute arrival-time shift is not useful for the synthetic-generation
  purpose this project needs (§1); a smaller, more clearly differentiated
  *k* is preferable even if a metric like inertia would keep improving
  with more clusters. This step should also be validated by inspecting
  each cluster's summary statistics (mean/median commute distance,
  arrival time, vehicle availability, etc.) discussed in `methodology.md`.

## 7. How clusters will be used downstream

```
employee_features.parquet
        │  (this document: preprocessing → clustering)
        ▼
  cluster assignments
  (each employee row labeled with an archetype, e.g. cluster 0-k)
        │
        ▼
  synthetic employee sampling
  (draw a cluster per synthetic employee according to real cluster
   proportions, then sample each feature from that cluster's fitted
   within-cluster distribution — not the population-wide marginal —
   so correlations observed within a cluster, e.g. long commutes
   co-occurring with earlier arrivals, are preserved)
        │
        ▼
  workplace charging demand scenarios
  (each synthetic employee's cluster-derived arrival/departure time
   defines their charging window, dwell time bounds session length,
   commute distance/VMT determines energy needed, vehicle availability
   determines whether the vehicle is present to charge — EV
   penetration and fuel-type assumptions, deferred per
   docs/feature_engineering_plan.md §3, are then overlaid on top)
```

Concretely:

1. **Cluster assignments** attach a categorical label to every row in
   `employee_features.parquet` (via the `HOUSEID`+`PERSONID` key), turning
   the flat feature table into a labeled dataset — this is the direct
   output of §5/§6 and requires no new data beyond what's already in the
   processed table.
2. **Synthetic employee sampling** treats each cluster as a template: a
   synthetic workforce of arbitrary size is generated by first sampling a
   cluster (weighted by that cluster's share of the real employee
   population, to preserve realistic archetype mix), then sampling each
   feature — commute distance, arrival time, dwell time, vehicle
   availability, demographics — from that specific cluster's fitted
   distribution (or, for a GMM, directly from its fitted per-component
   Gaussian). This is why clusters need to capture *joint* structure, not
   just be a discretization of one feature: sampling from within-cluster
   distributions is what lets a long commute distance still co-occur with
   an earlier arrival time and a longer dwell time in the generated data,
   reproducing correlations that independent per-feature sampling from the
   whole population's marginals would destroy.
3. **Workplace charging demand scenarios** consume the synthetic
   employees produced in step 2: each one's arrival/departure time defines
   a charging window, dwell time bounds the maximum session length,
   commute distance/VMT determines energy consumed, and vehicle
   availability determines whether a vehicle is even present to plug in.
   EV penetration rate and vehicle fuel-type assumptions are layered on
   top of this synthetic population at that stage (as already scoped out
   of feature engineering in `docs/feature_engineering_plan.md` §3) —
   nothing about the clustering or sampling steps needs to change when
   those scenario parameters are varied.
